What's the Problem with Missing Documentation?

Not many companies really care about documentation. And not many of them know how much lack of thereof costs. And I'm not talking just about money, what's more important is its impact on the error margin especially on critical things especially in new hires. Is your Achilles' heel exposed?

Disclaimer: When I talk about documentation, I mean internal documentation of software products. There should be documentation for more than software products, but that's not within the scope of this article.

On-Boarding Time

The first problem is obvious: on-boarding of every new hire takes more time, hence it costs more money. On top of that, the new guy will be distracting his colleagues with questions and distractions are really, really bad for creative professions.

Now I'm not arguing against asking questions, I'm merely saying that if those are questions every new developer will ask, these just should be documented.

It might be hard to quantify and it's non-obvious where the time is wasted. It ranges from relatively easy to figure out questions such as "Where do I start?" or "How do I run the tests?" to "It doesn't behave as it should, what am I missing?" and "How is our server set up, I need to change XY?" which in my experience is where most of time is unnecessarily wasted.

Highly Increased Error Margin Especially on Critical Errors

So that's one thing, losing money AND time which you could use to get ahead of your competition. Still, that's not the worst. If a developer have to make assumptions, nasty things happen.

And what if your only senior developer who knows the project in-and-out leaves?

Why Don't We Write Documentation?

We're lazy, that's why. Typically developers write documentation in Markdown format, then they have to commit the changes and push them to GitHub. Then, they go to GitHub and preview that everything looks good. At least that's what I do and what I've seen others to do. That's too many steps for something essential.

I believe by simplifying the workflow, we can make it more natural for developers to write documentation. (Keep reading, I have a solution!)

What Types of Documentation Do I Need to Care About?

High level documentation is what I talked about until now. Things like dependencies of our app, how to run tests, how to deploy it to staging and production, server configuration and so on and so forth. But there's another important and equally overlooked type of documentation I'd like to mention and that is what I call process documentation.

Process documentation is documenting our thinking process. Often we find something to be unintuitive and oftentimes the reason can be tracked to some either legacy or not obvious reason. Many especially younger developers tend to change this sort of "problems" only to realise what reason did it have in the first place.

This is done primarily through issue trackers, pull requests on GitHub and commit messages.

The documentation you don't need to care about, in my opinion, is API documentation. I'm mentioning it because that's what many developers imagine when you say documentation and some of them will insist that it's important. For a web application, it probably isn't.
